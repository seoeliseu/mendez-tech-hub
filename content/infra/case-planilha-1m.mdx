## O problema

O usuário sobe uma planilha (CSV/XLSX) por um portal. O serviço precisa processar **1.000.000 de linhas**, validar, **persistir no banco** e liberar a **consulta** das linhas cadastradas — tudo em **até 20 minutos**, com **testes comprovando** o SLA. Pontos sensíveis: como **receber/gravar** um arquivo grande sem derrubar o servidor, e como **ler** sem estourar a memória.

## Orçamento de tempo — por que 20min é folgado

1.000.000 linhas em 1.200s exigem só **~833 linhas/s**. Parsing em streaming faz dezenas de milhares por segundo; `bulk insert` faz 50k–100k/s. Um único worker resolve em **~2–5 min** — os 20min são margem enorme; paralelismo vira segurança, não necessidade.

<Callout type="warn">
**Nunca passe o arquivo grande pelo app server.** Gravar 100+ MB no disco/memória da API é o erro clássico — OOM, disco cheio, timeout de upload. Na AWS o arquivo vai **direto pro S3** via **presigned URL**: o portal pede uma URL assinada à API, e o **browser** sobe o arquivo direto no S3. O app nunca toca o binário.
</Callout>

## Cada peça, passo a passo — o que é e o que faz

Cada caixa do diagrama explicada: o que a peça **é** (fora do case) e o que ela **faz aqui**.

### 1. Portal (browser) — onde o upload começa

**O que é.** A tela web onde o usuário escolhe a planilha. Roda no navegador do usuário, não no servidor.

**No case.** Antes de enviar o arquivo, pede à API uma *presigned URL* e faz o `PUT` do arquivo **direto no S3** com ela — o binário nunca passa pelo backend.

### 2. Presigned URL — a chave temporária

**O que é.** Uma URL comum do S3 + uma **assinatura** na query string que carrega a permissão embutida (ex: "pode fazer `PUT` neste objeto até 15:40"). Gerada com as credenciais da API, mas quem a usa (o browser) **não precisa de credencial AWS**.

**No case.** Autoriza o browser a subir **um** arquivo específico, por tempo curto. Expira → link morto. É o que torna seguro o upload direto do browser pro S3.

### 3. S3 (Simple Storage Service) — o armazém de arquivos

**O que é.** *Object storage* da AWS: guarda arquivos (objetos) dentro de *buckets*, endereçados por uma *key* (ex: `jobs/abc/planilha.csv`). Não é disco nem banco — é key → blob via HTTP. Escala pra petabytes, durabilidade de 99.999999999% (11 noves).

**No case.** Recebe a planilha via `PUT` direto do browser (multipart pra arquivos grandes). O arquivo **fica aqui**; o worker lê depois. Zero risco de encher disco de servidor.

### 4. EventBridge — o roteador de eventos

**O que é.** Um *event bus* (barramento de eventos) serverless. Serviços e apps **publicam eventos** nele; você cria **regras (rules)** que casam um padrão de evento e o entregam a um ou mais **targets** (SQS, Lambda, Step Functions...). É pub/sub gerenciado — o "carteiro" que ouve "aconteceu X" e avisa quem se importa. (Era o antigo CloudWatch Events.)

**No case.** O S3 emite o evento `Object Created` (payload = **metadados**: bucket, key, tamanho — nunca o arquivo). Uma rule casa esse evento e o entrega na fila SQS. É o que dispara o processamento **sozinho**, sem ninguém ficar perguntando "já chegou?".

### 5. SQS (Simple Queue Service) — a fila de trabalho

**O que é.** Fila de mensagens gerenciada. Um lado **põe** mensagens, o outro **puxa** (poll) e processa. Desacopla produtor de consumidor: se o consumidor cai ou está lento, as mensagens **esperam** na fila em vez de se perder.

**No case.** Guarda o job (`bucket` + `key`) até um worker pegar. Absorve picos (100 uploads juntos → 100 msgs na fila). Garante *at-least-once*; se o worker morre no meio, o *visibility timeout* devolve a msg pra fila; falha crônica cai na **DLQ**.

### 6. ECS Fargate — onde o worker roda

**O que é.** Um jeito de rodar *containers* sem gerenciar servidor. **ECS** é o orquestrador de containers da AWS; **Fargate** é o modo em que a AWS provisiona a CPU/RAM sob demanda (você não cuida de máquina EC2). "Serverless pra containers".

**No case.** Roda o worker que consome a SQS e processa a planilha. Escolhido em vez de **Lambda** porque Lambda tem teto de **15 min** de execução e o SLA é 20 — Fargate roda o tempo que precisar.

### 7. Aurora / RDS — o banco

**O que é.** Banco **relacional** gerenciado. **RDS** roda Postgres/MySQL gerenciados pela AWS (backup, patch e réplica automáticos). **Aurora** é a versão própria da AWS, compatível com Postgres/MySQL, mais rápida e com storage que cresce sozinho.

**No case.** Persiste o 1M de linhas via *bulk* `COPY` (não INSERT 1-a-1). Depois serve a consulta paginada (índice + paginação por cursor) das linhas cadastradas.

### 8. CloudWatch — os olhos

**O que é.** A stack de observabilidade nativa da AWS: **métricas**, **logs** e **alarmes**. Todo serviço AWS já emite métrica pra ele.

**No case.** Mede duração do job, taxa de falha e tamanho da fila SQS. Um alarme dispara se o job passar do tempo ou a DLQ encher.

<Callout type="tip">
**Mini-glossário dos termos que aparecem:** *multipart upload* = subir o arquivo em pedaços paralelos e retomáveis. *Visibility timeout* = tempo que a msg fica "invisível" depois que um worker a pega; sem *ack*, ela reaparece na fila. *DLQ (dead-letter queue)* = fila-lixeira pras msgs que falharam N vezes. *Bulk insert / COPY* = gravar milhares de linhas numa tacada só. *Paginação por cursor (keyset)* = paginar por "depois do id X" em vez de `OFFSET` grande (que fica lento em 1M linhas).
</Callout>

## Código

A API só assina a URL; o browser faz o PUT direto no S3:

```csharp
// API: gera a URL assinada; o browser faz o PUT direto no S3
var req = new GetPreSignedUrlRequest
{
    BucketName = "uploads-planilhas",
    Key = $"jobs/{jobId}/{fileName}",
    Verb = HttpVerb.PUT,
    Expires = DateTime.UtcNow.AddMinutes(15),
};
string url = s3.GetPreSignedURL(req); // devolve pro portal — o app nao ve os bytes
```

O worker lê o S3 como stream e insere em lotes — a memória não cresce com o arquivo:

```csharp
// Worker: le o S3 como STREAM e insere em lotes — memoria nao cresce com o arquivo
using var resp = await s3.GetObjectAsync("uploads-planilhas", key);
using var reader = new StreamReader(resp.ResponseStream);
using var csv = new CsvReader(reader, CultureInfo.InvariantCulture);

const int BATCH = 10_000;
var buffer = new List<Linha>(BATCH);

await foreach (var linha in csv.GetRecordsAsync<Linha>())
{
    buffer.Add(Validar(linha));
    if (buffer.Count >= BATCH)
    {
        await BulkInsertAsync(buffer);          // COPY / SqlBulkCopy
        await AtualizarProgresso(jobId, BATCH);
        buffer.Clear();                          // solta o lote — nunca segura 1M na RAM
    }
}
if (buffer.Count > 0) await BulkInsertAsync(buffer);
```

## Comprovando o SLA — teste de carga (k6)

Gera a planilha de 1M linhas, roda o fluxo ponta a ponta e falha se passar de 20min:

```javascript
import http from "k6/http";
import { check, sleep } from "k6";

export const options = {
  scenarios: { um_arquivo: { executor: "shared-iterations", vus: 1, iterations: 1, maxDuration: "25m" } },
  thresholds: { iteration_duration: ["p(95)<1200000"] }, // 1.200.000 ms = 20 min
};

export default function () {
  const jobId = subirPlanilha1M();              // presigned PUT de 1M linhas no S3
  const inicio = Date.now();
  let status = "processing";
  while (status === "processing") {
    const r = http.get(`https://api.exemplo.com/jobs/${jobId}`);
    status = r.json("status");
    check(r, { "sem erro": (x) => x.status === 200 });
    if (Date.now() - inicio > 20 * 60 * 1000) break; // guarda dura de 20min
    sleep(5);
  }
  check(status, { "processou em ate 20min": (s) => s === "done" });
}
```

<Callout type="tip">
**Prove em 3 níveis.** *Unitário* no parser/validador (Vitest/xUnit). *Integração* com Testcontainers (Postgres real) medindo o tempo do bulk insert de 1M linhas. *Carga* k6 ponta a ponta com o threshold de 20min acima — esse é o que vale como prova do SLA no CI.
</Callout>

</Callout>
